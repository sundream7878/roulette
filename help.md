# 네이버 카페 댓글 수집 시스템 - 현재 상황 정리

## 🔍 문제 분석 결과

### 실제 상황
- **웹 페이지에서 보이는 댓글**: 44개
- **API가 반환하는 댓글**: **12개만**
- **수집된 참가자**: 12명 (정확함)

### 핵심 발견사항

**네이버 카페의 모든 공개 API가 전체 댓글을 제공하지 않습니다.**

테스트한 API 목록:
1. ✗ `cafe-articleapi/v2` → 12개만 반환
2. ✗ `cafe-articleapi/v2.1` → 500 에러
3. ✗ Article Detail API → 500 에러  
4. ✗ `CommentList.json` → API 존재하지 않음

**결론**: 현재 사용 가능한 공개 API로는 전체 44개 댓글을 가져올 수 없습니다.

## 🚫 왜 API가 일부만 반환하나요?

네이버 카페는 다음과 같은 이유로 API를 제한합니다:

1. **보안 정책**: 로그인하지 않은 사용자에게는 제한된 데이터만 제공
2. **서버 부하 방지**: 대량의 데이터 요청 차단
3. **크롤링 방지**: 자동화된 데이터 수집 제한


## 🧹 데이터 섞임 및 초기화 문제 해결 (2.13 업데이트)

많은 시도 끝에 데이터가 계속 섞이거나 이전 글의 데이터가 보이는 현상을 방지하기 위해 다음 장치들을 마련했습니다.

### 1. 주요 원인 및 조치 사항
- **URL 정규화 (Canonical URL)**: 네이버 카페는 동일한 글도 주소 형태가 여러 가지(모바일, PC, 별명 등)일 수 있습니다. 이를 모두 하나의 표준 형태로 변환하여 DB에 저장함으로써 데이터가 분산되거나 섞이는 것을 원칙적으로 차단했습니다.
- **세션 초기화 (Session Clear)**: 새로운 수집을 시작할 때마다 내부 통신 세션을 초기화하여 이전 게시글의 상태가 전이되지 않도록 했습니다.
- **디버깅용 응답 ID (Response ID)**: 이제 모든 서버 응답에는 고유 ID가 부여됩니다. 브라우저 개발자 도구(F12) 콘솔에서 `DEBUG: API Response Received` 로그를 통해 현재 화면에 보이는 데이터가 언제 생성된 것인지 정확히 확인할 수 있습니다.

### 2. 문제가 지속될 때 확인 체크리스트
1. **서버 재시작**: `comment_dart.py`가 실행 중인 터미널에서 `Ctrl+C` 후 다시 실행하여 최신 로직(정규화 등)이 활성화되도록 합니다.
2. **웹 페이지 새로고침 (F5)**: 브라우저 캐시 문제를 방지하기 위해 새로고침을 수행합니다.
3. **'✨ 새 이벤트 생성' 버튼 사용**: 이 버튼은 UI와 입력값을 완전히 비우고 새로운 수집을 위해 준비하는 안전한 시작점입니다.
4. **브라우저 콘솔 확인**: F12를 눌러 `response_id`가 매번 바뀌는지, 그리고 `Total: 6`과 같이 서버 로그와 일치하는지 확인합니다. 만약 서버 로그는 6인데 화면에 21이 나온다면, 브라우저가 이전 데이터를 메모리에 들고 있는 것이므로 강력 새로고침(Ctrl+F5)이 필요합니다.

### 3. 기술적 상세 로그
- `monitor_view.py`: `fetch_comments`와 `load_comments` 모두에서 URL 표준화 로직 적용 완료.
- `scraper.py`: `get_new_comments` 시작 시 세션 강제 클리어 로직 추가.

### 요약

수동으로 새로고침(F5)을 눌러야만 **총 댓글 작성자 수**와 **확정 참여자 명단**이 바뀜.
네이버 댓글 자동 수집 → DB 저장 → 화면 자동 반영 흐름이 중간에 끊겨 있는 상태.

### 단계별 문제 분석

#### [단계 1] 네이버 댓글 수집 (백그라운드 스레드)
- **기대**: 모니터링 시작 후 백그라운드 스레드가 주기적으로 Naver API를 폴링하여 새 댓글을 DB에 자동 저장
- **현실**: `start_background_monitoring()` 내부에서 `datetime` 모듈 미임포트로 스레드가 시작 즉시 crash됨 (→ `from datetime import datetime` 추가로 조치 완료)
- **확인 방법**: `monitor_debug.log`에 `Polling Naver...` 로그가 주기적으로 쌓이는지 확인

#### [단계 2] Supabase DB 저장
- **기대**: 백그라운드 스레드가 새 댓글을 감지하면 Supabase `commenters` 테이블에 즉시 upsert
- **현실**: 단계 1이 crash되므로 수동으로 `/api/fetch_comments`를 POST 요청할 때만 저장됨
- **확인 방법**: Supabase 대시보드에서 `commenters` 테이블에 새 행이 추가되는지 확인

#### [단계 3] Socket.IO 브로드캐스트 (Supabase 폴링)
- **기대**: `_supabase_poll_loop` 이 2초마다 `commenters` 테이블을 확인하고, 이전과 달라지면 `update_participants` 이벤트를 모든 클라이언트에 emit
- **현실**: 단계 2에서 데이터가 들어오지 않으면 감지 자체가 불가능 (폴링은 작동 중)
- **확인 방법**: 브라우저 콘솔(F12)에서 `Socket Update received:` 로그 확인

#### [단계 4] 모니터 페이지(`monitor.html`) 화면 갱신
- **기대**: Socket 이벤트 수신 시 페이지 새로고침 없이 총 댓글 수, 참여자 명단 자동 업데이트
- **현실**: 단계 3에서 emit이 없으면 갱신 없음. emit이 있어도 빈 데이터면 화면 변화 없음
- **조치 완료**: `window.location.reload()` 제거 후 DOM 직접 갱신 방식 적용

#### [단계 5] 룰렛 페이지(`index.html`) 자동 새로고침
- **기대**: 참여자 데이터 변경 감지 시 페이지 자동 새로고침으로 최신 명단 반영
- **현실**: 이전 수정에서 `reload()` 대신 DOM update 방식으로 교체되어 새로고침 안 됨
- **조치 완료**: `update_participants` 이벤트 수신 시 게임 미진행 상태이면 `window.location.reload()` 실행하도록 복구함

### 핵심 병목 요약

| 단계 | 상태 | 비고 |
|------|------|------|
| 네이버 댓글 자동 수집 | ❌ 수동 트리거만 동작 | 백그라운드 스레드 crash 수정 완료, Render 재배포 필요 |
| Supabase 저장 | ⚠️ 수동 수집 시에만 | 단계 1 해결 시 함께 해결 |
| Socket emit | ✅ 작동 | DB에 데이터가 있을 때 |
| 모니터 페이지 갱신 | ✅ 작동 | Socket emit이 올 때 |
| 룰렛 페이지 새로고침 | ✅ 복구 완료 | Socket emit이 올 때 |

**결론**: 근본 원인은 **백그라운드 수집 스레드 안정성** 문제. Render 서버에 재배포 후 `monitor_debug.log`를 확인하여 스레드가 정상 작동하는지 검증 필요.

### 임시 해결책 (지금 당장 쓸 수 있는 방법)
- 모니터 페이지에서 **"수집 시작"** 버튼을 누를 때마다 Naver에서 최신 댓글을 가져와 DB에 저장되며, 이후 Socket을 통해 모니터/룰렛 페이지가 자동 갱신됨
- 즉, **수집 버튼만 눌러주면** 수동 F5 새로고침은 필요 없어질 수 있음 (단, 버튼을 주기적으로 눌러야 하는 불편함은 있음)
